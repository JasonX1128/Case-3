# DS235H Case 3 Vehicle Routing Solver

This repository contains a Python/Gurobi solver for the DS235H Case 3 vehicle routing problem. The model reads the case workbook, applies the operational constraints in `constraints.txt`, and produces vehicle routes with service times, loads, costs, feasibility checks, progress traces, incumbent reports, and bound plots.

The main entry point is `vrp_gurobi.py`. It supports two solving approaches:

- **Arc-flow MIP**, the default solver. This builds a compact mixed-integer programming model with vehicle symmetry breaking, automatic MIP-start persistence, progress logging, incumbent report refreshes, and bound plots.
- **Route-based branch-and-price**, enabled with `--route-based` or `--branch-and-price`. This builds feasible route columns, solves set-partitioning master problems, prices new routes, branches on route relationships, and can checkpoint/resume long searches.

## Requirements

Use Python 3.11 or newer. The repository has been run with a local virtual environment in `.venv`, but no checked-in requirements file is currently present.

Required Python packages:

```bash
pip install gurobipy openpyxl
```

Recommended optional packages:

```bash
pip install numpy matplotlib
```

The optional packages improve solve/diagnostic quality:

- `numpy` allows the solver to use faster pricing helpers in the route-based path.
- `matplotlib` is required for saving MIP bound plots under `plots/`.

You also need a working Gurobi license. A local `gurobi.lic` works through Gurobi's normal discovery process. For Gurobi Web License Service, copy `.env.example` to `.env` and fill in:

```text
GUROBI_WLSACCESSID=
GUROBI_WLSSECRET=
GUROBI_LICENSEID=
```

The solver looks for these values in environment variables or in a `.env` file beside `vrp_gurobi.py`.

## Input Workbook Contract

The solver expects the workbook supplied with `--workbook` to contain these sheets:

- `Delivery Stops`
- `Vehicle Fleet`
- `Depot`
- `Distance Matrix (miles)`
- `Travel Time Matrix (min)`
- `Cost Parameters`

The loader expects the workbook to follow the case template:

- `Delivery Stops`: stop ID, customer name, address, demand in kg, earliest time, latest time, and service minutes.
- `Vehicle Fleet`: vehicle ID, vehicle type, capacity, cost per mile, fixed daily cost, available-from time, and available-until time.
- `Depot`: depot open and close times.
- Distance and travel-time matrices: square matrices covering depot node `0` plus every stop.
- `Cost Parameters`: named scalar costs, including fuel, fuel consumption, driver wage, and late-delivery penalty.

The implementation currently assumes `Early_Delivery_Penalty = 0`. If the workbook sets a nonzero early-delivery penalty, the solver raises an error rather than silently solving the wrong objective.

## Business Rules Modeled

The hard constraints are centralized near the top of `vrp_gurobi.py` and mirrored in `constraints.txt`:

1. Stop 3, Redstone Medical Center, must be served between 12:45 PM and 5:15 PM.
2. Stops 18, 19, 20, 21, and 22 are subject to the Hillcrest Road five-ton weight limit and may only be served by Small Vans.
3. Stops 7 and 8, Redstone Plaza Mall, must be served by the same vehicle on consecutive stops, in either order, with one combined 45-minute service block.
4. Stop 14, Sunrise Senior Living, cannot be served before 10:00 AM.
5. Main Street construction is always active; 20 minutes are added to any travel leg to or from Stops 1 through 6.
6. A driver may not make deliveries during the 11:30 AM to 1:30 PM lunch window unless the route includes one 30-minute break scheduled inside that window. Routes with all delivery service outside the lunch window do not require a break.
7. Vehicle 5 / Driver 5 may not service northern district Stops 23 through 28.
8. Medium Truck 3 has a faulty liftgate. If Vehicle 3 serves a stop with demand over 50 kg, 15 minutes of service time are added.
9. Stops 10, 11, 12, and 13 are subject to the Oakwood noise ordinance and may only be served between 8:00 AM and 7:00 PM.
10. Stop 27, Redstone Elementary School Cafeteria, must be served before 4:00 PM.

Every printed and saved route report ends with a constraint check. Treat `Constraint check: passed` as the quick sanity marker that the final route set satisfies these modeled rules.

## Objective

The solver minimizes total route cost. At a high level, route cost combines:

- fixed daily cost for each used vehicle,
- distance cost based on vehicle cost per mile plus fuel cost,
- driver labor cost, billed by whole active hours,
- late-delivery penalties when service starts after a workbook latest time.

The solver reports both total objective value and route-level cost components. The default arc-flow report includes drive, service, billed hours, and distance cost by vehicle. The route-based report includes active minutes, distance, route cost, lunch break timing when present, and per-stop service timing.

## Quick Start

From the repository root:

```bash
python3 vrp_gurobi.py --workbook Team_02_Data.xlsx --time-limit 300 --progress
```

This runs the default arc-flow MIP for up to five minutes, prints progress, writes an objective trace CSV beside the workbook, saves an incumbent report, and writes a MIP bound plot under `plots/`.

For a shorter incumbent-hunting run using the bundled Gurobi profile:

```bash
python3 vrp_gurobi.py \
  --workbook Team_02_Data.xlsx \
  --mip-profile two-minute \
  --progress
```

For a longer solve with no user-facing time or gap cap:

```bash
python3 vrp_gurobi.py --workbook Team_02_Data.xlsx --unlimited --progress
```

## Arc-Flow MIP Usage

The arc-flow MIP is the default path and is generally the best first option when you want a direct Gurobi model with robust incumbent discovery.

Common options:

```bash
python3 vrp_gurobi.py \
  --workbook Team_02_Data.xlsx \
  --time-limit 900 \
  --mip-gap 0.05 \
  --mip-profile hybrid \
  --progress \
  --progress-interval 30
```

Useful arc-flow flags:

| Flag | Meaning |
| --- | --- |
| `--time-limit SECONDS` | Overall solve time cap. |
| `--mip-gap GAP` | Relative optimality gap target, such as `0.05` for 5%. |
| `--mip-profile NAME` | Applies a profile from `gurobi_mip_profiles.json`. Current profiles include `hybrid`, `two-minute`, `max-accuracy`, and `find-incumbent`. |
| `--mip-persist-dir PATH` | Overrides where reusable `.mst`/`.sol` starts are stored. |
| `--removed-vehicles ID...` | Removes vehicles from the fleet before solving, for scenario testing. Accepts space-separated or comma-separated IDs. |
| `--progress` | Prints throttled progress updates during long solves. |
| `--objective-trace PATH` | Writes objective/bound snapshots to a custom CSV path. |
| `--no-objective-trace` | Disables objective trace output. |
| `--incumbent-report-path PATH` | Writes the human-readable incumbent report to a custom path. |

The default MIP persistence directory is:

```text
.mip_persist/<workbook-stem>/
```

If a prior solve wrote a reusable MIP start, later arc-flow runs automatically load it. The solver also appends its built-in constructive seed as a fallback start when possible.

## Branch-and-Price Usage

Use the route-based path when you want the set-partitioning branch-and-price implementation:

```bash
python3 vrp_gurobi.py \
  --workbook Team_02_Data.xlsx \
  --route-based \
  --time-limit 900 \
  --progress \
  --checkpoint-path Team_02_Data_route_based_checkpoint.json
```

Common branch-and-price options:

| Flag | Meaning |
| --- | --- |
| `--route-based` or `--branch-and-price` | Enables the route-based solver. |
| `--max-cg-iterations N` | Limits column-generation iterations at each tree node. Use `0`, `none`, or `unlimited` for no cap. |
| `--pricing-time-limit SECONDS` | Per-pricing solve time limit. |
| `--pricing-columns-per-type N` | Maximum improving routes added per vehicle type per pricing iteration. |
| `--pricing-workers N` | Number of pricing worker processes. Use `1` to disable parallel pricing or `auto` for automatic selection. |
| `--dual-stabilization-alpha VALUE` | Dual smoothing weight between `0` and `1`; `1.0` uses raw duals. |
| `--dual-stabilization-mode adaptive|fixed` | Controls whether stabilization changes with dual movement. |
| `--max-bp-nodes N` | Caps processed branch-and-price nodes. |
| `--checkpoint-path PATH` | Saves resumable branch-and-price state to JSON. |
| `--resume-from PATH` | Resumes from a saved branch-and-price checkpoint. |

Resume a branch-and-price run with:

```bash
python3 vrp_gurobi.py \
  --workbook Team_02_Data.xlsx \
  --route-based \
  --resume-from Team_02_Data_route_based_checkpoint.json \
  --progress
```

When resuming, do not combine `--resume-from` with `--removed-vehicles`; the checkpoint may not encode the removed-fleet scenario.

## Scenario Testing

To test whether the routes remain feasible after removing vehicles:

```bash
python3 vrp_gurobi.py \
  --workbook Team_02_Data.xlsx \
  --removed-vehicles 3 5 \
  --time-limit 600 \
  --progress
```

Equivalent comma-separated syntax is supported:

```bash
python3 vrp_gurobi.py --workbook Team_02_Data.xlsx --removed-vehicles 3,5
```

Vehicle IDs are validated against the workbook, and the solver refuses to remove every vehicle.

## Reading the Incumbent Report

Each vehicle section includes:

- vehicle ID and vehicle type,
- depart and return times,
- total route load versus capacity,
- active time, drive/service time or distance, and billed hours where available,
- lunch break timing when one is required,
- distance and route costs,
- route stop IDs,
- route details with customer names and addresses,
- per-stop service start times,
- early/late minutes depending on solver path.

The route IDs omit the depot for readability. Internally, every route starts and ends at depot node `0`.

## Interrupts and Long Runs

The solver installs interrupt handlers so long runs can be stopped without losing the most useful artifacts.

On `Ctrl+C` or a termination signal, the solver attempts to retain:

- the objective trace,
- the latest incumbent report,
- the latest branch-and-price checkpoint, if enabled,
- a partial arc-flow bound plot, if available,
- a reusable arc-flow MIP start, if available.

The summary file `Team_02_Data_interrupt_artifacts.txt` lists the retained artifact paths from the most recent interrupted run.

## Tuning Notes

Start with the default arc-flow MIP plus a time limit:

```bash
python3 vrp_gurobi.py --workbook Team_02_Data.xlsx --time-limit 300 --progress
```

Then try:

- `--mip-profile two-minute` for fast incumbent hunting.
- `--mip-profile hybrid` for a balanced practical run.
- `--mip-profile max-accuracy` for longer proof-focused work.
- `--unlimited` when you want Gurobi or branch-and-price to continue until it naturally finishes or is interrupted.
- `--removed-vehicles` for sensitivity analysis.
- `--route-based --checkpoint-path ...` for resumable branch-and-price experiments.

The file `gurobi_mip_profiles.json` is intentionally simple. You can add new named profiles by adding a top-level JSON object with a `description` string and `params` object containing Gurobi parameter names and values.

## Native Pricing Helper

`native_pricing_helper.c` is optional acceleration for route-based pricing. If NumPy is installed and the C source exists, the solver compiles it automatically into `.native_cache/` using the `CC` environment variable or the system `cc`.

If compilation fails, the solver falls back to pure Python/NumPy-compatible logic. You do not need to manually build this helper for normal use.

## Troubleshooting

### Gurobi license errors

Confirm that Gurobi is installed and licensed:

```bash
python3 -c "import gurobipy as gp; print(gp.gurobi.version())"
```

If using WLS credentials, verify that `.env` has all three required fields:

```text
GUROBI_WLSACCESSID
GUROBI_WLSSECRET
GUROBI_LICENSEID
```

### Missing workbook sheet or bad matrix size

The workbook must keep the expected sheet names and matrix dimensions. The distance and travel-time matrices must include depot node `0` plus every delivery stop.

### No bound plot is saved

Install `matplotlib`:

```bash
pip install matplotlib
```

### Route-based run is slow

Try lowering `--pricing-columns-per-type`, setting a `--pricing-time-limit`, reducing `--max-cg-iterations`, or using `--pricing-workers auto`. For long experiments, always set `--checkpoint-path` so progress can be resumed.

### Arc-flow run starts from stale artifacts

The solver normally handles saved starts automatically. If you want a completely fresh arc-flow run, use a new `--mip-persist-dir` path or move the old `.mip_persist/<workbook-stem>/` directory aside.
